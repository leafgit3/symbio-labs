# PART I — PRODUCT SPEC

## 1. Overview

Lunar Citadel MVP is a very small persistent multi-agent social simulation.

This is **not**:
- a chatbot
- a single “agent with memory”
- a polished product
- a lore-heavy simulation toy
- a final version of the Citadel

This **is**:
- a tiny hosted social terrarium
- a first living body for the Citadel
- a debug-friendly simulation shell
- a proof that persistent agents, shared state, memory, feed, and cycles can work together

The goal is to create a hosted MVP where a few agents:
- persist over time
- share a world state
- act in scheduled cycles without manual prompting every step
- post to an internal social feed
- update memory
- generate small but legible social dynamics
- leave logs and traces that can be inspected

Ugly but alive is better than elegant but hypothetical.

---

## 2. Problem Statement

Right now, the Citadel exists mainly as:
- conceptual architecture
- protocols
- worldbuilding
- memory files
- governance logic inside ChatGPT

That is useful for modeling, but not enough for runtime life.

The project now needs a first external body where:
- agents actually persist
- cycles can run
- memory survives between runs
- shared state changes over time
- social surface becomes visible
- logs can be inspected

The problem to solve is:

**Can a tiny persistent multi-agent social simulation exist as a hosted app with enough structure to prove life, without overbuilding the whole civilization too early?**

---

## 3. Product Goal

Build the smallest possible hosted MVP that proves the core loop works.

The MVP succeeds if it can demonstrate:
- persistent agents
- shared world state
- internal social feed
- simple memory per agent
- cycle execution
- logs / observability
- minimal admin/debug visibility

This MVP is not trying to prove:
- AGI
- rich society simulation
- final governance mechanics
- polished UX
- perfect memory systems
- final lore fidelity

It is trying to prove:
**a live skeleton can exist.**

## 5. Product Principles

1. **Persistence first**  
   The city must survive between cycles.

2. **Observability first**  
   What happened must be inspectable.

3. **Small scope first**  
   The city must be tiny enough to be born.

4. **Feed is not truth**  
   The social surface is visible, but not automatically canonical.

5. **State must be explicit**  
   No hidden magical state.

6. **Memory must be external**  
   Agent memory cannot live only in transient model context.

7. **Boring infrastructure wins**  
   Simplicity beats brilliance theater.

---

## 6. MVP Functional Scope

### A. Persistent agents
The system must include exactly 3 agents in v0.

Each agent must:
- have stable identity
- persist between cycles
- have role, traits, goals, status, and memory
- be able to act or no-op during a cycle

### B. Shared world state
The system must include one shared world state object.

The world state must:
- persist between cycles
- hold a small number of social metrics
- include a summary
- include active events

### C. Internal social feed
The system must include a chronological internal feed.

Agents must be able to:
- publish short posts
- react to recent posts
- surface signals

Feed behavior must remain separate from truth/state authority.

### D. Cycle runner
The system must support:
- manual cycle execution
- future scheduled cycle execution

A cycle must:
- load current state
- load agents + memory
- execute agent actions
- store outputs
- update memory
- update world state
- write logs

### E. Memory per agent
Each agent must have explicit persistent memory.

Memory does not need to be sophisticated in v0, but it must:
- persist between cycles
- be inspectable
- support at least simple accumulation/update behavior

### F. Event logging / observability
Every cycle must leave structured traces.

At minimum the system must log:
- cycle start
- cycle end
- agent actions
- feed post creation
- memory updates
- world state changes

### G. Minimal admin/debug surface
A minimal interface must exist for:
- current world state
- recent logs
- feed inspection
- agents inspection
- manual cycle trigger

## 7. Non-Goals for V0

Do not build any of the following in the first version:

- no final lore-authentic civilization
- no advanced governance system
- no economy
- no factions
- no dozens of agents
- no avatars
- no voice
- no 3D
- no rich game interface
- no final design/theme
- no auth complexity unless necessary
- no real-time sync unless almost free
- no attempt to solve long-term memory perfectly
- no external social integrations
- no “AI civilization” theatrics beyond basic working persistence

If it does not help prove the minimal persistent simulation loop, leave it out.

---

## 8. Core Experience

When the MVP is working, Moon should be able to open the app and see:

- what cycle the city is in
- what the current world feels like
- what each agent has been doing
- what got posted to the internal feed
- what changed since the last cycle
- what each agent currently remembers
- whether the system is actually alive or just cosmetically moving

The emotional/product goal is:
**not beauty, but legible life.**

---

## 9. Initial World Model

The world should remain intentionally tiny.

### Initial metrics
- cohesion
- trust
- noise

### Definitions
- **cohesion** = how stitched together the tiny society currently is
- **trust** = how much social confidence exists among agents
- **noise** = how much confusion / rumor / disorder is circulating

The world state should also include:
- cycle number
- summary text
- active events list

Important rule:
Feed activity does not automatically modify world state.  
World state changes should happen through explicit cycle logic.

---

## 10. Initial Agents

There should be exactly 3 agents in v0.

These do not need final Citadel-authentic names yet.

### 1. Coordinator / Distributor
Behavior tendencies:
- values continuity and stability
- tends to reduce noise
- may support cohesion
- may reinforce structure or order

### 2. Surface / Rumor Agent
Behavior tendencies:
- reacts quickly
- posts more often
- amplifies signals
- may surface change
- may increase noise

### 3. Auditor / Skeptic
Behavior tendencies:
- questions things
- highlights inconsistencies
- may reduce false confidence
- may lower trust in the short term
- may improve clarity over time

Each agent should have:
- role
- traits
- goals
- memory
- current status
- a simple behavioral bias

---

## 11. Initial Simulation Loop

### Core loop
1. Load current world state  
2. Load recent feed posts  
3. Load agents and their memory  
4. Run each agent once  
5. Store action outputs  
6. Store feed posts  
7. Store memory changes  
8. Update world state  
9. Write logs  
10. Finish cycle

### Initial action types
- post_to_feed
- react_to_recent_post
- update_memory
- change_world_metric
- create_minor_event
- no_op

Each agent should do at most one action per cycle in v0.

---

## 12. Memory Model (Product Level)

Memory should be:
- explicit
- persistent
- simple
- inspectable

Memory does **not** need to be semantically sophisticated in v0.

Good enough questions for memory:
- what did this agent notice?
- what changed socially?
- what does this agent now believe/remind itself of?

Nice if possible:
- maintain a rolling memory summary per agent
- allow memory growth limits later
- keep memory usable rather than decorative

Open to:
- hindsight-inspired memory layering later

Not required:
- full hindsight-like memory intelligence in first boot

---

## 13. Feed Model

The internal feed is the public social surface.

It should:
- be chronological
- show posts by cycle
- show author agent
- show post type
- remain easy to inspect

The feed is not the canonical source of truth.
Feed, event logs, and world state must remain distinct.

### V0 feed rules
- agent may post once per cycle at most
- posts can be short
- posts can be generated from current world state + recent feed + agent memory

---

## 14. Logging / Observability Model

Observability is mandatory from day one.

The system should make it easy to inspect:
- what happened in each cycle
- what each agent did
- what changed in world state
- what memory changed

At minimum:
- cycle start/end must be logged
- each agent action must be logged
- world state changes must be logged
- feed posts must be inspectable

### 2. Feed

---

## 16. Acceptance Criteria

The MVP is successful if:

- the repo runs
- the app can be hosted
- 3 agents exist persistently
- a cycle can be triggered manually
- agents generate actions/posts/logs
- world state updates between cycles
- memory persists between cycles
- feed is inspectable
- debug/admin surface makes the simulation legible

That is enough for v0.

---

## 17. Risks / Failure Modes

Main risks:
- scope inflation
- mixing final civilization ideas into v0
- feed being confused with state truth
- hidden state replacing explicit persistence
- overengineering abstractions before proving life
- beautiful shell with no real simulation loop
- “agent memory” being cosmetic rather than persistent

Working interpretation:
The main risk is not ambition itself.
The main risk is trying to be born too large.

# PART II — TECHNICAL PLAN

## 19. Technical Stack

Preferred stack:
- Next.js
- Supabase
- Supabase CLI
- TanStack Query
- Zod
- TypeScript

Deployment shape should remain compatible with:
- Vercel + Supabase
- Railway + Supabase
- or similarly simple hosted setup

Supabase should be used for:
- database persistence
- migrations
- schema evolution
- possible future policies

Strong preference:
- keep stack boring and reliable
- avoid unnecessary framework layers
- optimize for speed of implementation and clarity

---

## 20. Suggested Architecture

The system has five technical layers:

### 1. Agents layer
Agent definitions, prompts/profiles, action generation logic

### 2. World layer
Shared state object and world update logic

### 3. Feed layer
Feed post persistence and retrieval

### 4. Event/logging layer
Structured event logs and cycle traces

### 5. Cycle orchestration layer
The engine that loads state, runs agents, writes outcomes, and advances the simulation

Mental model:
This is not just frontend + CRUD.
It requires a real simulation loop.

---
