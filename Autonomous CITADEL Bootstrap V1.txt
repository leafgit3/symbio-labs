1. PROJECT INTENT
━━━━━━━━━━━━━━━━━━━━

This project is not a chatbot, not a single “agent with memory,” and not a lore-heavy simulation toy.

It is a very small persistent multi-agent social simulation prototype.

The goal is to create a hosted MVP where a few agents:
- persist over time
- share a world state
- act in scheduled cycles without manual prompting every step
- post to an internal social feed
- update memory
- generate small but legible social dynamics
- leave logs and traces that can be inspected

This is the first bootable body for a larger conceptual project called “Lunar Citadel,” but this repo should only implement the smallest useful version.

This first version should optimize for:
- simplicity
- speed of implementation
- external persistence
- inspectability
- future extensibility

This first version should NOT optimize for:
- aesthetics
- theming
- final architecture
- maximum abstraction
- lore completeness
- visually rich simulation
- “AGI vibes”

Ugly but alive is better than elegant but hypothetical.

2. DESIGN PHILOSOPHY
━━━━━━━━━━━━━━━━━━━━

Core principle:
Build the smallest possible persistent social simulation that proves the loop works.

The MVP should feel like:
- a tiny social terrarium
- a living dashboard
- a debug-friendly simulation shell

Not like:
- a polished product
- a game
- a complex agentic framework
- a worldbuilding-heavy fiction interface

The app should make it possible to answer:
- Who are the agents?
- What happened last cycle?
- What is the current shared world state?
- What did each agent remember?
- What changed over time?
- What did the agents post or react to?

5. CORE IDEA
━━━━━━━━━━━━━━━━━━━━

The application represents a tiny persistent society.

Agents exist over time, interact through a shared feed, remember things, and slightly influence a shared world state.

This is not a single conversational AI.
It is a multi-agent simulation loop.

Mental model:

world state
↓
agents read state
↓
agents act
↓
feed + logs update
↓
memory update
↓
world state update
↓
next cycle

6. APPLICATION LAYERS
━━━━━━━━━━━━━━━━━━━━

The system has five essential layers:

1. Agents
Persistent simulated actors with identity, goals, traits, memory, and behavior.

2. World state
A small shared state object representing the current condition of the micro-society.

3. Feed
Public social surface where agents post/react.

4. Logs / events
Structured record of what happened.

5. Cycle execution
A repeatable process that advances the simulation.

━━━━━━━━━━━━━━━━━━━━
7. MVP FUNCTIONAL SCOPE
━━━━━━━━━━━━━━━━━━━━

The MVP should include exactly these core capabilities:

A. Persistent agents
- 3 agents only
- stable identity over time
- role, traits, goals, memory, current status

B. Shared world state
- one shared state object
- minimal metrics
- simple event list
- updated every cycle

C. Internal social feed
- agents can publish short posts
- feed is chronological
- posts are part of observable simulation surface
- posts are not automatically “truth”

D. Cycle runner
- manual cycle trigger first
- scheduled cycle support second
- each cycle loads state, runs agents, stores outputs, updates world

E. Memory per agent
- each agent should persist some memory
- memory should survive between cycles
- memory should be inspectable
- memory can be simple at first

F. Event logging / observability
- every cycle should generate logs/events
- recent logs should be visible in admin/debug view

G. Minimal admin/debug UI
- inspect world state
- inspect agents
- inspect feed
- inspect events
- trigger cycle manually

10. INITIAL WORLD MODEL
━━━━━━━━━━━━━━━━━━━━

Keep world state intentionally tiny.

Initial global metrics:
- cohesion
- trust
- noise

Definitions:
- cohesion = how stitched together the tiny society currently is
- trust = how much social confidence exists among agents
- noise = how much confusion / rumor / disorder is currently circulating

World state should also have:
- brief summary text
- active events list
- current cycle number

Important:
Feed posts do not automatically modify world state.
World state changes should happen through explicit cycle logic.

━━━━━━━━━━━━━━━━━━━━
11. INITIAL AGENTS
━━━━━━━━━━━━━━━━━━━━

Start with exactly 3 agents.

They do not need final Citadel-authentic names yet.

Suggested roles:

1. Coordinator / Distributor
Behavior:
- values order, continuity, and shared state stability
- tends to reduce noise
- may support cohesion

2. Surface / Rumor Agent
Behavior:
- reacts quickly
- posts more often
- amplifies signals
- may increase noise or surface hidden shifts

3. Auditor / Skeptic
Behavior:
- questions things
- highlights inconsistencies
- may reduce false confidence
- may sometimes reduce trust in the short term

Each agent should have:
- a small prompt profile
- a behavioral bias
- capacity to post
- capacity to remember
- capacity to affect the world slightly

━━━━━━━━━━━━━━━━━━━━
12. INITIAL CYCLE LOGIC
━━━━━━━━━━━━━━━━━━━━

The cycle runner should stay simple.

Basic cycle flow:

1. Start cycle
- create CycleRun record
- load latest WorldState
- load recent FeedPosts
- load agent records + memory

2. For each agent
- generate one action or no-op
- optionally generate one feed post
- optionally generate one memory update
- store outputs

3. Aggregate results
- append EventLogs
- compute simple world state changes
- produce updated world summary

4. Persist
- save new/updated WorldState
- save EventLogs
- save FeedPosts
- save memory updates
- update agent statuses

5. Finish cycle
- close CycleRun
- make results visible in admin/debug UI

V0 action types:
- post_to_feed
- react_to_recent_post
- update_memory
- change_world_metric
- create_minor_event
- no_op

Important:
One action per agent per cycle is enough.

━━━━━━━━━━━━━━━━━━━━
13. MEMORY MODEL FOR V0
━━━━━━━━━━━━━━━━━━━━

Memory should be explicit and simple.

Do not attempt perfect semantic memory in v0.

V0 memory behavior:
- agents can store short memory entries
- memory persists between cycles
- each cycle may add 0 or 1 memory entry per agent
- memory can later be compacted, but compaction is not required for first boot

Good enough memory questions per cycle:
- what did this agent notice?
- what changed socially?
- what does this agent now “believe” or remember?

Nice if possible:
- maintain a small rolling memory summary per agent
- cap memory growth later if needed

Open to using inspiration from Hindsight-style memory layering,
but not required to build Hindsight-like sophistication now.

━━━━━━━━━━━━━━━━━━━━
14. FEED MODEL
━━━━━━━━━━━━━━━━━━━━

The internal social feed is important because it makes the society visible.

It should:
- be chronological
- show posts by cycle
- show author agent
- show post type
- be easy to inspect

The feed is the public/social surface.
It is not the source of canonical truth.
WorldState and EventLog remain distinct from the feed.

V0 feed behavior:
- agents may post once per cycle at most
- content can be short
- posts can be generated from current world state + recent feed context + agent memory

━━━━━━━━━━━━━━━━━━━━
15. EVENT / LOGGING MODEL
━━━━━━━━━━━━━━━━━━━━

Logs are mandatory from day one.

The app should make it easy to inspect:
- what happened in each cycle
- what each agent did
- what changed in world state
- what memory got updated

At minimum:
- every cycle start/end should be logged
- every agent action should be logged
- every world state change should be logged
- every feed post should be visible

The point is not beauty.
The point is observability.

━━━━━━━━━━━━━━━━━━━━

Important distinction:
This app is not just frontend + CRUD.
It needs an actual simulation loop.

So backend orchestration should be treated as first-class.

Priority order:
1. define schemas
2. persist objects
3. implement cycle runner
4. connect UI
5. add scheduled execution

If needed, manual cycle execution is enough before scheduled jobs are fully wired.

22. SCHEDULING
━━━━━━━━━━━━━━━━━━━━

- cycles can be scheduled automatically
- manual trigger exists no matter what

V0 scheduling requirement:

or

23. LLM / AGENT LOGIC INTEGRATION
━━━━━━━━━━━━━━━━━━━━

LLM integration should be modular.

The LLM should be used for:
- agent action generation
- feed post generation
- short memory updates
- maybe world summary generation

The LLM should NOT be treated as:
- hidden persistent system state
- sole source of memory
- sole source of truth

Structured outputs are strongly preferred.

If possible:
- agent outputs should map to known action schema
- avoid freeform blob logic as much as possible
